## 🎯 핵심 키워드

<aside>
💡 주요 내용들을 정리하고, 여러분만의 생각을 통해 정리해보세요!

- 레퍼런스를 참고하여 정의, 속성, 장단점 등을 적어주셔도 괜찮습니다.
- 조사는 공식 홈페이지 **Best**, 블로그(최신 날짜) **Not Bad**
</aside>

- CI/CD
    - 지속적 통합(Continuous Integration)과 지속적 배포/전달(Continuous Delivery/Deployment)
    - CI와 CD 과정을 하나의 자동화된 워크플로, 즉 '파이프라인'으로 구성함
- GitHub Actions
    - 코드 저장소(repository)에서 CI/CD(지속적 통합/지속적 배포) 파이프라인을 자동화하는 플랫폼
    - 리포지토리에서 발생하는 이벤트(예: 코드 푸시, 풀 리퀘스트)에 따라 빌드, 테스트, 배포 등 복잡한 워크플로를 자동으로 실행 가능함
- Reverse Proxy
    - 클라이언트의 요청을 받아서 웹 서버에 전달하고, 응답을 받아 클라이언트에게 다시 전달하는 서버
    - 보안 강화, 성능 향상, 부하 분산 등의 이점을 제공하며, 클라이언트에게는 웹 서버의 실제 IP 주소가 노출되지 않도록 함
- HTTPS
    - HTTPS는 '**하이퍼텍스트 전송 프로토콜 보안**(Hypertext Transfer Protocol Secure)'의 약자로, 웹 브라우저와 웹사이트 간 통신을 암호화하여 안전하게 보호하는 통신 규약입니다. 이는 HTTP의 보안이 강화된 버전으로, 사용자의 로그인 정보, 결제 정보 등 민감한 데이터가 가로채이거나 조작되는 것을 막아줍니다.

## ⚠️ 스터디 진행 방법

**1. 스터디 전 준비**

- 핵심 키워드와 미션을 모두 수행해 주세요.

**2. 스터디 시간**

    - 각자가 정리한 핵심 키워드와 미션을 서로 공유해 주세요.
    - 파트장(또는 스터디장)은 파트원들의 수행 내용을 보고 피드백할 부분을 정리하여 전달해 주세요.

## ✨ 위클리 스크럼 과제

---

위클리 스크럼 과제는 OO주차 위클리 스크럼 아이스 브레이킹과 공통 질문을 모두 작성한 후, 아래에 질문을 복사 붙여넣기 한 다음 다같이 작성해주시면 됩니다. 

1. JWT는 `Header`, `Payload`, `Signature` 세 부분으로 나뉩니다. 이 중 `Payload`에는 유저의 식별 정보(`userId`, `email` 등)를 담는데요. 이곳에 유저의 **'비밀번호'나 '주민등록번호' 같은 민감한 정보**를 담으면 안 되는 이유는 무엇일까요? (힌트: `jwt.io` 사이트의 기능을 떠올려 봅시다:>)
2. 구글 클라우드 콘솔 설정에서 **`승인된 리디렉션 URI`** (http://localhost:3000/oauth2/callback/google) 를 아주 정확하게 등록해야만 로그인이 동작했습니다. 만약 구글이 이 URI 검사를 하지 않고, 요청하는 대로 아무 주소로 리디렉션 시켜준다면 해커가 어떤 식으로 악용할 수 있을까요?
