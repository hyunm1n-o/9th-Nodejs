## 🎯 핵심 키워드

<aside>
💡 주요 내용들을 정리하고, 여러분만의 생각을 통해 정리해보세요!

- 레퍼런스를 참고하여 정의, 속성, 장단점 등을 적어주셔도 괜찮습니다.
- 조사는 공식 홈페이지 **Best**, 블로그(최신 날짜) **Not Bad**
</aside>

- Swagger
    - RESTful API 문서화 및 테스트 도구
    - API 명세를 시각화하여 웹 내에서 바로 테스트 가능하게 해주는 오픈소스 프레임 워크
    - 프론트 개발자 좋아해요 ^^
    - YAML 파일을 통해 API 명세를 별도로 정의하고 이를 바탕으로 Swagger UI를 구성함!
- OpenAPI
    - Open Application Programming Interface, 공개 API
    : 누구나 사용할 수 있도록 공개된 API 를 의미
    - 개발 시 시간과 비용 절감과 동시에 퀄리티 좋은 서비스 개발이 가능하기 때문에 사용한다
    - API 사용을 위해 공개키를 발급 받아야만 제공된 데이터를 모두 이용 가능하다
- OpenAPI Component
    - API의 재사용 가능한 독립적인 부분
    - OpenAPI Specification에서 정의된 API의 구성 요소(스키마, 파라미터, 응답 등)을 컴포넌트 라고 부름
    - 장점 : 재사용성, 표준화, 유지보수성, 독립적인 모듈

## ✨ 위클리 스크럼 과제

---

위클리 스크럼 과제는 OO주차 위클리 스크럼 아이스 브레이킹과 공통 질문을 모두 작성한 후, 아래에 질문을 복사 붙여넣기 한 다음 다같이 작성해주시면 됩니다. 

1. 워크북 마지막 부분에서는 `errorCode`를 분리하는 이유(예: U001, U002)가 **프론트엔드 협업** 때문이라고 설명했습니다. 만약 여러분이 프론트엔드 개발자라면, 백엔드가 이렇게 '친절한' 에러를 내려줄 때와 '500 서버 오류'만 내려줄 때, 각각 어떤 점이 다를 것 같은지 상상해서 이야기해 봅시다.
    - 프론트에서 에러를 알려줬을 때 바로 어떤 서버 에러인지 바로 알아차릴 수 있을 거 같아 해결이 더 빠를 거 같다. 그리고 친절한 에러가 오면 어느정도 짐작이 되기 때문에 마냥 답답하지 않을 듯…
2. `index.js` 파일에서 미들웨어의 **등록 순서**는 매우 중요합니다. 만약 `app.use(express.json())` 미들웨어를 모든 라우터(예: `app.post('/api/v1/users/signup', ...`) 보다 **아래에** 등록한다면 어떤 문제가 발생할까요? Express가 요청을 처리하는 흐름을 생각하며 답해 보세요.
    - 데이터를 제대로 파싱하지 못해서 읽을 수 없는 문제가 생길 거 같음
3. **(에러 데이터와 보안)** 7주차 `DuplicateUserEmailError` 예제는 에러 응답(`res.error`)의 `data` 필드에, 회원가입 시도에 실패한 **사용자의 요청 정보(`email`, `name`, `address`, `phoneNumber` 등) 전체**를 담아 반환했습니다. 이렇게 반환하는 방식이 디버깅할 때는 이 `data` 덕분에 "어떤 값으로 요청했길래" 에러가 났는지 정확히 알 수 있어서 편리했는데요. 하지만, 실제 운영(배포) 환경에서 에러 응답에 `email`이나 `phoneNumber` 같은 **민감한 개인정보**를 그대로 담아 반환하는 것은 어떤 **보안 문제**를 일으킬 수 있을까요? (해시된 비밀번호를 포함한 경우, 그 정보가 노출되는 것은 어떨까요?) 디버깅의 편의성과 보안 사이에서, 에러 응답의 `data` 필드에는 **어디까지** 정보를 담는 것이 적절하다고 생각하나요?
    - 잘못하면 사용자의 개인정보 유출이기 때문에 큰 문제가 생길 거 같다… 비밀번호도 디코딩을 어떻게든 해내는 사람이 있지 않을까?! 싶다.
